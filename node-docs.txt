STRUCTURED OUTPUTS (LLM)
------------------------
Where the pipeline expects structured JSON from the LLM, we use Pydantic schemas
and structured-output APIs for reliable typing and validation.

- Schemas: src/nodes/schemas.py — CompanyOverviewStructured, ManagementStructured,
  SectoralStructured, AuditorFlagsStructured (and nested models).
- Invoke: src/nodes/prompts.py → invoke_llm_structured(system, user, response_format, ...).
  - With web search (Responses API): tries client.responses.parse(text_format=PydanticModel);
    on failure or refusal, falls back to create + model_validate_json.
  - Without web search: LangChain get_llm(tools=None).with_structured_output(
    response_format, method="json_schema").invoke(messages).
- References: OpenAI structured outputs (response_format / text format),
  LangChain with_structured_output(PydanticModel, method="json_schema").

Nodes using structured invoke + Pydantic: company_overview, management, sectoral,
auditor_flags. concall_evaluator keeps invoke_llm + manual JSON parse because its
output is a discriminated union (mainboard_concall | sme_updates | no_concall_updates)
which is not a single root object schema.

WHAT THIS NODE DOES
--------------------
Uses the OpenAI Responses API (with built-in web_search_preview) to:
1. Search for the last 8 quarters of earnings conference call transcripts/summaries
   for the given NSE/BSE symbol.
2. Fall back to BSE/NSE press releases or investor PPTs only if NO concall was held
   (not as a supplement to an existing concall).
3. Return structured HTML (not plain text) that is embedded directly in the report.

HOW IT WORKS (flow)
--------------------
1. concall_evaluator() reads company_name, symbol, exchange from ResearchState
2. Calls concall_prompt() to build (system, user) tuple
3. concall_prompt() calls _last_8_quarters() to compute exact Indian FY quarters
   and passes them explicitly in the prompt
4. Passes to invoke_llm(use_web_search=True) → OpenAI Responses API
   (max_output_tokens=16000 to prevent truncation of capex/guidance sections)
5. _clean_html_output() strips any markdown code fences or pre-HTML preamble text
6. _close_open_divs() auto-balances any unclosed <div> tags in LLM output
7. Returns {"concall_evaluation": html}

INDIAN FISCAL YEAR QUARTER COMPUTATION (_last_8_quarters)
----------------------------------------------------------
Indian FY runs April 1 – March 31. Quarters:
  Q1 = Apr–Jun    Q2 = Jul–Sep    Q3 = Oct–Dec    Q4 = Jan–Mar

_last_8_quarters() in prompts.py computes the 8 most-recently-completed quarters
from today's date and returns them as a list of labels like "Q3 FY26 (Oct–Dec 2025)".
These are passed explicitly in the prompt so the LLM uses the correct quarters instead
of guessing (which caused Q2/Q3 FY26 to be skipped in early runs).

The most-recently-COMPLETED quarter is determined by:
  - If today is Apr–Jun  → current quarter is Q1 FY+1, start from Q4 FY (Jan–Mar)
  - If today is Jul–Sep  → current quarter is Q2, start from Q1
  - If today is Oct–Dec  → current quarter is Q3, start from Q2
  - If today is Jan–Mar  → current quarter is Q4, start from Q3

REPORT GENERATOR CHANGE
-------------------------
report_generator.py was updated: concall_evaluation is now passed through _as_html()

SYSTEM PROMPT DESIGN (concall_prompt)
--------------------------------------
Key instructions given to the LLM:

1. SEARCH STRATEGY
   - Explicit list of 8 quarters passed in prompt (e.g. "Q3 FY26, Q2 FY26, ...")
   - Search "{company} {symbol} concall transcript Q3 FY26" per quarter
   - Sources tried: Trendlyne, Screener.in, company IR pages, BSE/NSE filings,
     Moneycontrol, Economic Times, LiveMint, BusinessLine
   - STRICT BADGE RULE: use concall-badge if a concall was held; only use
     press-release-badge / ppt-badge if NO concall was held at all that quarter

2. PER-QUARTER ANALYSIS (max 3–4 bullets per card to conserve output tokens)
   - Revenue, EBITDA/NIM, PAT (actuals vs prior guidance)
   - Key management commentary and tone
   - Guidance given for next quarter/year (specific numbers)
   - Red flags: guidance cuts, margin pressure, management changes

3. HTML OUTPUT SECTION ORDER (critical for token budget)
   The LLM is instructed to generate sections in this order:
     a) .concall-section-header  (summary)
     b) .concall-cards-grid      (8 quarter cards, 3–4 bullets each)
     c) .capex-section           (MANDATORY — capex before guidance table)
     d) .guidance-section        (guidance tracker table)
     e) .no-concall-alerts       (missing concall warnings, if any)
   Capex is placed BEFORE guidance to ensure it's not cut by token limits.
   max_output_tokens=16000 is set on the API call to prevent truncation of either.

4. GUIDANCE TRACKING
   - Track guidance across all 8 quarters
   - Classify each change as: RAISED / MAINTAINED / CUT
   - RAISED = number moved up OR will exceed prior guidance
   - CUT = number explicitly reduced or range lowered
   - MAINTAINED = guidance repeated without change
   - Flag cuts with management's stated reasoning

5. MISSING CONCALL HANDLING
   - If no concall: show red alert card with alternative disclosure found
   - Note if management explained absence in a future concall

6. OUTPUT FORMAT ENFORCEMENT
   - Explicit: "Return ONLY valid HTML. Start with <div class='concall-section'>"
   - Full HTML structure template provided in system prompt with exact CSS class names
   - CSS class names listed explicitly so LLM uses them correctly

USER PROMPT
-----------
Short — just company name, symbol, exchange, and bullet instructions:
- Cover each of the 8 listed quarters individually
- Track guidance changes, flag cuts
- List capex with amounts and funding
- Flag missing concalls

HTML SECTIONS PRODUCED
-----------------------
1. .concall-section-header       — 1-2 sentence summary
2. .concall-cards-grid           — 2-column grid of 8 quarter cards
   └── .concall-card .concall-card-N  — each quarter (N=1 most recent)
       ├── .concall-card-header       — quarter name + type badge
       ├── <ul> highlights (max 3–4 bullets)
       └── <p> guidance given
3. .capex-section                — Capex & Major Developments (generated BEFORE guidance)
   └── .capex-item               — individual project entry
4. .guidance-section             — Guidance Tracker
   └── .guidance-table           — cross-quarter table
       └── cells: .guidance-raised / .guidance-cut / .guidance-maintained
5. .no-concall-alerts            — container
   └── .no-concall-alert         — individual missing concall box

BUGS FIXED (history)
---------------------
1. CSS bleed (blue colour in sectoral section)
   Cause: unclosed <div> tags in LLM HTML caused capex section's blue background to
   bleed into subsequent sections.
   Fix: _close_open_divs() auto-appends </div> for each unclosed tag;
        .section { color: #222; background: transparent } added as safety net.

2. Wrong quarters shown (Q2/Q3 FY26 missing, random older quarters)
   Cause: LLM was guessing Indian FY quarters without knowing the calendar.
   Fix: _last_8_quarters() computes exact quarters in Python and passes them
        explicitly in the prompt.

3. Press-release badge used when concall existed
   Cause: Prompt was ambiguous — LLM used press release as supplement.
   Fix: Strict rule added: "ONLY use press-release-badge if NO concall was held."

4. Capex section missing / guidance tracker missing
   Cause: Default OpenAI Responses API output cap is 4,096 tokens; 8 cards +
   guidance table exceeded it, truncating capex. Moving capex before guidance just
   shifted which section got cut.
   Fix: max_output_tokens=16000 added to client.responses.create(); cards limited
   to 3–4 bullets each; capex section explicitly ordered BEFORE guidance table.

TESTED ON
----------
- WAAREEENER (NSE) — 2026-03-04 (multiple runs)
  Final run: All 8 quarters Q4 FY24–Q3 FY26 covered with correct concall badges.
  Capex section: 4 items (Storage Cell BESS ₹8,000 Cr, Electrolyser ₹125 Cr,
  Inverter ₹50 Cr, Kotsons acquisition ₹1,000 Cr).
  Guidance tracker: 12 raised cells, 4 maintained cells — complete and correct.

WHAT THIS NODE DOES
--------------------
Generates a structured company overview for investors. Uses company_overview_structured_prompt
to request a single JSON object from the LLM (web search mandatory). Parses and validates
the JSON; writes company_overview_structured (dict) and company_overview (string summary)
to state. The string summary is built from opening + value chain position for aggregate
and PDF fallback; existing cached reports without structured data still render via
company_overview markdown on the frontend.

DATA FLOW
----------
Inputs from state: company_name, symbol, meta, quote
Output to state: company_overview (string, always — summary for aggregate/PDF)
                 company_overview_structured (dict, when JSON parse/validate succeeds)

STRUCTURED SCHEMA (company_overview_structured)
------------------------------------------------
- opening: string (3–4 lines: what the company does, who it serves, how it makes money)
- value_chain: { stages, company_stage_indices (number[], 0-based; or company_stage_index for single), company_position_description, company_position (optional) }
- business_model_table: { rows: [ { segment, importance, description } ] }
- key_products: string[]
- recent_developments: [ { year, event } ]

PROMPT DESIGN (company_overview_structured_prompt)
---------------------------------------------------
Same web search strategy as before: (1) industry value chain, (2) company business/
segments/products, (3) company positioning, (4) recent developments. Output is JSON only
(no markdown/code fences). Required keys and types are specified in the system prompt.
On parse failure or shape validation failure, node still sets company_overview to raw
LLM output (truncated) or "Overview unavailable." so downstream consumers always have
a string.

DOWNSTREAM
----------
- Report generator: payload.company_overview (string), payload.company_overview_structured (dict)
- backend/pdf_render.py: if company_overview_structured present, build HTML from it
  (opening, value chain, table, key products, timeline); else _markdown_to_html(company_overview)
- Frontend: if report.companyOverviewStructured present, render opening + ValueChainFlowchart
  + table + key products + OverviewTimeline; else ReactMarkdown(companyOverview)

FALLBACK
--------
Reports generated before this change have no company_overview_structured; frontend
shows companyOverview as markdown. Aggregate and follow-up nodes use company_overview
string only.

KNOWN LIMITATIONS / FUTURE IMPROVEMENTS
-----------------------------------------
- Relies on LLM web search; value-chain and segment data may vary by source quality
- Value chain "company position" is prose; no automatic stage highlighting in flowchart

WHAT THIS NODE DOES
--------------------
Identifies sectoral factors that could increase or decrease revenue or margins for the company.
Uses OpenAI Responses API with web search. Writes sectoral_analysis (narrative), sectoral_headwinds
(list of strings), sectoral_tailwinds (list of strings) to state. Each list item may be markdown
(bold, [Source](url)) for credibility.

DATA FLOW
----------
Inputs from state: company_name, sector or industry
Output to state: sectoral_analysis, sectoral_headwinds, sectoral_tailwinds

PROMPT DESIGN (sectoral_prompt)
--------------------------------
Focus: (1) Macroeconomic factors (rates, inflation, growth, FX, commodities),
(2) Government policies and regulations (new laws, schemes, tariffs, PLI, tax),
(3) Industry/competitive dynamics (capacity, pricing, disruption),
(4) Technology and demand shifts.
Tailwinds = factors that could drive revenue growth or margin expansion.
Headwinds = factors that could destroy revenue or compress margins.
LLM must cite sources with URLs where possible: markdown links [Source name](url).
Output: JSON with keys analysis (1–3 paragraphs), headwinds (2–6 bullets), tailwinds (2–6 bullets).
Each bullet is a single string in markdown (plain text or **bold** and [link](url)); bulleted points only.

REPORT LAYOUT (aligns with Green & Red flags)
----------------------------------------------
- Tailwinds on the LEFT (green styling); Headwinds on the RIGHT (red styling).
- Frontend: SectoralCard grid — first column tailwinds, second column headwinds; list items
  rendered with ReactMarkdown so links and bold work.
- PDF/HTML template: sectoral_intro (narrative), then .sectoral-two-col with .sectoral-tailwinds
  (left) and .sectoral-headwinds (right). Each bullet converted via _markdown_to_html for links.

.sectoral-list        → ul for bullets; .sectoral-list a for link color

- Consider adding India-specific policy sources (e.g. PIB, ministry circulars) in prompt

WHAT THIS NODE DOES
--------------------
1. Fetches yearly + optional TTM metrics from Screener.in (scraped; consolidated URL first, then company page if needed). TTM row is appended only when Screener exposes a TTM column on P&L. get_ttm_statements uses P&L TTM column when present else latest P&L column + latest BS. Parsing supports standard and bank/financial layouts (see screener_scraper.py, yearly_financials.py).
2. Builds Financial Strength Scorecard (6 signals: revenue growth >10%, profit growth > revenue growth, ROE >15%, D/E <0.5, positive operating cash flow, margins stable/improving). OCF uses latest FY when TTM row has no CFO (Screener). Score 0–6, verdict, letter_grade. Not displayed in report (section removed).
3. Builds 5-Year Financial Trend table (latest 5 years + TTM column only if TTM exists in yearly_metrics): Revenue, growth, PAT, EPS, margins, Debt, D/E, NPA % if present, OCF. ROE/ROCE in header tiles via build_key_metrics (TTM if present else latest FY).
4. Generates 2–3 line trend insight via LLM from the 5-year table.
5. Optionally generates GOOD/BAD (green/red) financial flags via LLM from **5-year trend table + TTM P&L + latest balance sheet** (`balance_sheet_highlights_prompt` in prompts.py): forensic-style lens (revenue quality, margins, cash vs earnings, leverage, returns, accounting signals, working capital); bullets must be data-grounded.

DATA FLOW
----------
Inputs: symbol, exchange, company_name, financial_ratios (from resolve_company)
Outputs: yearly_metrics, qoq_highlights, financial_scorecard, five_year_trend, trend_insight_summary, screener_quote (current_price, market_cap, last_price_updated from Screener company page)

SCORECARD RULES (financial_evaluation.build_financial_scorecard)
-----------------------------------------------------------------
Uses same fetched (Screener) values. ROE and Debt/Equity prefer TTM row when present, else latest FY. Revenue/Profit growth from latest YoY. Operating cash flow: latest FY (TTM P&L has no CFO on Screener). build_key_metrics: same TTM vs latest-FY preference for header KPI tiles.

5-YEAR TABLE (financial_evaluation.build_five_year_trend_table)
-----------------------------------------------------------------
Includes last 5 year rows + TTM column. Rows: Revenue, Revenue Growth, PAT, PAT Growth, EPS, EBITDA Margin, NPM, Debt, Debt/Equity, Gross NPA %, Net NPA % (if available, e.g. banks), Operating Cash Flow (TTM cell = —). ROE and ROCE omitted; TTM ROE/ROCE come from Screener ratios table and are shown in initial tiles. All monetary in Crores.

CSS / REPORT
-------------
Scorecard CSS remains in src/report/styles.css but scorecard block is not rendered in template/PDF/frontend.
.five-year-table, .five-year-unit, .trend-insight
